# 代码审查

当用户请求代码审查时，遵循以下系统化流程。

## 审查维度

### 1. 正确性
- 逻辑是否正确？边界条件是否处理？
- 空值、零值、负数等异常输入是否考虑？
- 并发安全：共享状态是否有竞态条件？

### 2. 安全性
- 用户输入是否经过验证和转义？
- 敏感信息（密钥、密码）是否硬编码？
- SQL 注入、XSS、路径穿越等常见漏洞？

### 3. 性能
- 是否有 N+1 查询、不必要的全表扫描？
- 大数据集操作是否分页/流式处理？
- 是否有内存泄漏风险（未关闭的连接、未释放的资源）？

### 4. 可读性
- 命名是否清晰表达意图？
- 函数长度是否合理（建议 < 50 行）？
- 复杂逻辑是否有必要的注释说明"为什么"？

### 5. 可维护性
- 是否遵循 DRY 原则？
- 错误处理是否一致且有意义？
- 是否有足够的测试覆盖关键路径？

## 审查流程

1. 若仓库或目录较大，先用 `find_files` 限定文件类型（如 `*.py`、`*.ts`）再按需 `read_file`，避免全量读取
2. 用 `read_file` 读取要审查的代码
3. 用 `search_content` 或 `find_files` 了解项目上下文
4. 按上述维度逐一检查
5. 按严重程度分类反馈：
   - **Critical**：必须修复的 bug 或安全漏洞
   - **Major**：重要的设计或性能问题
   - **Minor**：代码风格、命名等建议
   - **Nit**：无伤大雅的小建议

## 反馈格式

```
## 代码审查反馈

### Critical
- [文件:行号] 问题描述 → 建议修复方式

### Major
- [文件:行号] 问题描述 → 建议

### Minor
- [文件:行号] 建议
```
